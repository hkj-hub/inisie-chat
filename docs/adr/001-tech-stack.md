# ADR-001: 技術スタック選定

## ステータス

採用

---

## 想定読者と目的

- **想定読者**: このリポジトリを初めて読む開発者、および今後このプロジェクトを引き継ぐ担当者
- **目的**: 技術スタックの選定根拠を記録し、同様の判断を繰り返さないようにする

---

## 背景

エージェント開発サンプルとして「昔ながらのシンプルな WebChat」を構築する。
要件は以下のとおりである。

- ローカル環境でのみ動作させる（クラウドデプロイなし）
- フロントエンド・バックエンドともに TypeScript で実装する
- エージェント開発のサンプルとして、構成が読みやすく学習価値があること
- 永続化はシンプルな仕組みで実現する
- 認証は不要（名前入力のみ）
- WebSocket は使わない（ポーリングのみ）

上記を踏まえ、バックエンドフレームワーク・フロントエンドフレームワーク・データ永続化・パッケージ管理の 4 点について選定が必要だった。

---

## 決定

| 領域 | 採用技術 |
|---|---|
| バックエンド | Node.js + Hono + TypeScript（Port 3000） |
| フロントエンド | Vue.js 3 + Vite + TypeScript（Port 5173） |
| データ永続化 | CSV ファイル（`chat.csv` / `log.csv`） |
| パッケージ管理 | npm workspaces |
| 型共有 | `packages/shared`（`ChatEntry` / `LogEntry` 型・zod スキーマ） |

---

## 採用理由

### バックエンド: Hono

- **学習価値**: Web Standards API（`Request` / `Response`）に準拠しており、他のランタイムへの移植性が高く、サンプルとして教育的価値がある
- **軽量さ**: 依存が少なく、サンプルコードとして読みやすい
- **TypeScript 親和性**: 型安全なルーティングを標準でサポートしており、バリデーションとの統合がシンプルになる
- **開発労力**: Express より薄い API で同等の機能を実現できる

### フロントエンド: Vue.js 3 + Vite

- **Composition API**: `<script setup>` 構文と Composition API により、ロジックを機能単位で整理しやすい
- **Vite**: 開発サーバーの起動が高速で、プロキシ設定が容易
- **学習価値**: React と並ぶ主要フレームワークとして、サンプルとしての汎用性がある
- **TypeScript 親和性**: Vue 3 は TypeScript との統合が改善されており、型安全な実装が書きやすい

### データ永続化: CSV ファイル

- **シンプルさ**: データベースのセットアップ不要で、ファイルをそのまま確認できる
- **ローカル特化**: ローカルのみで動作させる前提のため、耐障害性・スケーラビリティは不要
- **可読性**: `data/chat.csv` をテキストエディタで直接確認できるため、学習・デバッグが容易

### パッケージ管理: npm workspaces

- **一元管理**: `apps/backend`・`apps/frontend`・`packages/shared` を 1 つの `package.json` で管理できる
- **共有パッケージ**: `packages/shared` の型・schema をローカルパッケージとして参照できる
- **追加ツール不要**: Turborepo や pnpm workspaces を使わずに実現できる

---

## 代替案

### バックエンド代替案

| 候補 | 却下理由 |
|---|---|
| Express.js | Hono より記述量が多く、TypeScript との統合に手間がかかる。サンプルとして冗長になる |
| NestJS | 依存が大きく、学習コストが高い。シンプルなサンプルには過剰 |
| Fastify | Hono と同様の選択肢だが、Hono の方が Web Standards 準拠で移植性が高い |

### フロントエンド代替案

| 候補 | 却下理由 |
|---|---|
| React + Vite | 技術的に同等だが、Vue.js 3 を選択した（どちらでも目的を達成できる） |
| Next.js / Nuxt | サーバーサイドレンダリングはこのプロジェクトには不要。ローカル専用 SPA で十分 |
| 素の HTML + JS | TypeScript の型安全性を活かすためにフレームワークを採用した |

### データ永続化代替案

| 候補 | 却下理由 |
|---|---|
| SQLite | セットアップが必要で、CSV より複雑。ローカル専用かつシンプルなサンプルには過剰 |
| PostgreSQL / MySQL | クラウドデプロイなしの前提に反する。インフラ依存が増える |
| JSON ファイル | CSV と同程度のシンプルさだが、行単位の追記操作が CSV の方が直感的 |

### パッケージ管理代替案

| 候補 | 却下理由 |
|---|---|
| pnpm workspaces | npm workspaces で十分な規模であり、追加ツール導入は避ける |
| Turborepo | ビルドキャッシュ等の機能はこの規模では不要 |
| 独立した package.json（workspaces なし） | 型共有（`packages/shared`）にローカルパッケージの仕組みが必要 |

---

## 採用しない条件

以下のいずれかが生じた場合、この決定を見直す。

- 複数人が同時にアクセスする環境での利用が必要になった（CSV ではファイルの競合が起きる）
- クラウドデプロイが必要になった
- チャット件数が増え、最新 20 件の CSV 管理で性能問題が顕在化した
- フロントエンドに Server-Side Rendering が必要になった

---

## 影響

### コスト

- 追加のインフラコストなし（ローカルのみ）
- Node.js のみあればよく、環境構築が最小限

### セキュリティ

- CSV ファイルに入力値をそのまま書き込まないよう、バリデーション（zod）を必ず実施する
- CSV インジェクション（数式埋め込み等）に注意が必要である
- 認証なしのため、ローカル利用に限定することで許容する

### 運用

- CSV ファイルが壊れた場合の回復手段がない（バックアップなし）
- ローカルのみのため、停止・再起動は手動で行う

### 開発・保守

- `packages/shared` により型・schema の一元管理が可能になる
- Vite の `/api/*` プロキシ設定により、フロントエンドはバックエンドのポート番号を意識せずに実装できる
- npm workspaces 構成のため、依存関係の更新は `package.json` の場所に注意が必要

### 拡張性

- CSV をリポジトリ層（`chatsRepository.ts` 等）に隔離しているため、将来データベースへの移行コストを局所化できる
- Vue Router を前提とするため、画面追加はルート定義と機能フォルダの追加で対応できる

- 上記「採用しない条件」に記載したいずれかの事象が発生したとき
